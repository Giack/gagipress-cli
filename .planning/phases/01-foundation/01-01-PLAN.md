Create `migrations/009_dashboard_rls_anon_select.sql` with the following SQL. The migration must:

1. Drop the existing `FOR ALL` policies (which allow anon INSERT/UPDATE/DELETE):
```sql
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON books;
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON content_ideas;
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON content_scripts;
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON content_calendar;
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON post_metrics;
DROP POLICY IF EXISTS "Enable all access for authenticated users" ON sales_data;
```

2. Re-create full access for the `authenticated` role (required for CLI operations which use the service key — but restoring explicit policies is a good practice):
```sql
CREATE POLICY "authenticated full access" ON books
  FOR ALL TO authenticated USING (true) WITH CHECK (true);
-- repeat for: content_ideas, content_scripts, content_calendar, post_metrics, sales_data
```

3. Add SELECT-only policies for the `anon` role (dashboard read access):
```sql
CREATE POLICY "anon read-only" ON books
  FOR SELECT TO anon USING (true);
-- repeat for: content_ideas, content_scripts, content_calendar, post_metrics, sales_data
```

Include a comment header explaining the purpose and which tables are covered. Use the exact SQL patterns from `01-RESEARCH.md` Code Examples section. Include all 6 tables (books, content_ideas, content_scripts, content_calendar, post_metrics, sales_data) — `content_scripts` is included for safety even though the dashboard does not query it in Phase 1.

Do NOT apply the migration — just write the file. The migration is applied manually via `supabase db push` after the file is committed.
